Getting Started

There are three steps to setting up your site with monitor404:

  1. Add your site to the monitor404 dashboard.
  2. Include the script on the pages you want to monitor.
  3. Configure which links on pages you want to be monitor. (optional – all are monitored by default.)

Adding a site

To add a site, go to the “Add Site” dialog and enter the domain name of your site.

Note that your site name should include a subdomain if it has one. For example, if your site is on www.reddit.com, include the “www”. The domain name you enter should be equal to the output of running window.location.host in JavaScript on your site.

Including the script

To monitor links on a page, include the following script anywhere on the page:

<script src="http://monitor404.s3.amazonaws.com/monitor404.min.js"></script>

The script requires jQuery, which you should also load somewhere on the page. It's tested with jQuery version 1.10.2 but will probably work with other versions.

That's it! monitor404 will check all links on which a user clicks for 404 errors. It will log errors in the dashboard and, by default, send you an email if it finds a broken link. If you only want to monitor certain kinds of links, refer to the next section.

Settings

monitor404 lets you customize which links it monitors using jQuery selectors and same-origin/different-origin rules. By default, monitor404 will check all links for 404 errors. However in certain cases, you may only wish to check a subset.

For example, you may already have logging or notifications set up on your server for HTTP 404 errors and only want to monitor links that go to external websites for errors. Or, you may run a blog and want to make sure all links in your posts stay working, but not really care if links in user-submitted comments break.

To change any of these settings, define a JavaScript object variable on window named _404_SETTINGS anywhere before $(document).ready() fires. (Note the leading underscore on the variable name because JavaScript doesn't allow variables to start with a numeral.)

Below is an example configuration using all settings, after which each setting is described in detail.

window._404_SETTINGS = {
    include: 'article a.ref', // only include links inside the <article> 
                              // element that have the class "ref"
    exclude: '.comments a', // exclude links in that are inside an element 
                            // with the class "comments"
    origin: 'different' // only monitor links that go to external sites
};

include

Default: 'a' (all links)
Format: string for a jQuery selector
Constraints: must select <a> elements only

include controls which links are monitored for 404 errors. If not defined, all links will be monitored. If defined, any selector you would use in jQuery (i.e. put inside a $()) is valid, as long as it selects link elements only. Trying to monitor links on non-link elements would be pointless, and it results in an error.

exclude

Default: '' (no links)
Format: string for a jQuery selector
Constraints: none

From the links that are included, exclude determines which to omit from monitoring. No links are excluded by default. As with include, any jQuery selector is valid.

origin

Default: 'both' (“internal” and “external” links)
Options: 'same', 'different', or 'both'

origin allows you to include/exclude links for monitoring based on whether they are internal or external; i.e. whether they link to another page on your site and have the 'same' hostname as the page from which they originate, or to a page off your site and have a 'different' hostname than the page from which they originate. By default, 'both' types are monitored.

Troubleshooting

Setting up monitor404 is quite straightforward, but things always have a tendency to go awry, don't they? The purpose of this section is to make troubleshooting any errors as painless as possible.

Getting Feedback

For every link check that monitor404 sends out, it returns a JSON response that contains some useful information.

To see this response, open the Chrome Developer Tools (I highly recommend Chrome though other major browsers have similiar solutions) and go to the “Network” tab.

If you click on a link normally, the new page will load before you can see the response from monitor404. To get around this, simply ctrl+click (or Apple+click, depending on your OS) on the link so that it opens in a new tab. In the “Network” panel on the first page, you should see a request to monitor404.com/api/check/.

If everything went well, you should see something like:

{result: 0, status: "success", message="Link queued for check."}

If this is the case, then yay! Everything's working – monitor404 recognized this request is associated with your site, got the link on which the user clicked, and has queued it for check (or retrieval from cache if it was checked recently).

If something went wrong, you may see something like:

{result: 1, error: "DoesNotExist", 
message: "UserSite matching host \"example.com\" was not found."}

If you got an error message, do not despair! Below is documented every error the monitor404 server may return to you, what the error means, and how to fix the issue.

Error Messages

KeyError: Required key "destination" not found in request.

This usually means the js script couldn't find the href property of the element on which the user clicked to send to the server. A common cause for this issue is that you're accidently including a non-link element, or the link element is missing an href property.

KeyError: Required key "source" not found in request.

This error means the URL of the page the user was on was not sent to the server. This error should not occur unless for some reason, running document.URL on the page containing the link returns undefined or null.

URLError: Could not parse origin header "[value]".

This error means either:

  1. The origin header the browser sent was malformed (i.e. not a parsable URL), or
  2. If [value] is None, the browser did not send an origin header.

The second is probably more likely to occur. Because the request made is cross-origin (originating from somewhere other than monitor404.com), the browser should send an Origin header. Check the headers of the Ajax request. If the Origin header is missing, it may be that the browser you're using doesn't support it.

DoesNotExist: UserSite matching host "[hostname]" was not found.

If you get this error, it means that monitor404 doesn't have a site in its database matching the hostname from which you're sending the request. To fix the error, add (or re-add) a site from the “Add Site” dialog with a Host that matches the host for which monitor404 is expecting as stated in the error message.

500 Internal Server Error

Uh oh! This means something went wrong on our end. Please contact support, and we'll get it fixed ASAP.

I didn't get any message whatsoever :(

The monitor404 server will always return a response. If there's no response, it probably means a request wasn't sent. Some common reasons for this are:

  1. The monitor404.min.js script wasn't included on the page.
  2. The link element on which you're clicking does not have a monitor404 click event bound to it.
  3. You have a JavaScript error on the page which is preventing the monitor404 script from working properly.

The solution to #1 is simple: make sure the script is included on the page. If it is, you can check #2 by seeing if you have a _404_SETTINGS variable defined to selectively include/exclude links on the page. If this variable is defined, check the jQuery selectors and/or origin rules you're using to make sure the desired links are included.

If both of the above check out, you may have a JavaScript error on the page. To check, open Chrome Developer Tools (or equivalent in other browsers), and go to the “Console” tab. Fix any JavaScript errors that are present there.

FAQ

I set everything up; how do I know if it's working?

Short answer: ctrl+click/Apple+click a link to open it in a new tab, check the Network panel in Chrome (or equivalent in other browsers), make sure the response is a happy success message.

Longer answer: see the “Troubleshooting” section.

How does monitor404 work?

When you set up monitor404, you include a tiny JavaScript file on pages you want to monitor. Whenever a user clicks a link that's being monitored, this JavaScript sends an Ajax request to a monitor404 server with the link they're trying to visit in the split second before they leave the page. monitor404 puts that link into an asyncronous processing queue, and then issues the URL a HEAD request to check the status code of the response.

Won't that Ajax request slow things down?

Yes, but not perceptibly. The request/response is so quick that it's virtually unnoticable, even if you're looking for it. Don't believe us? All the links on monitor404.com use monitor404. (why wouldn't they?) You tell me if it's negatively affecting your browsing experience.

So every time a user clicks on a link, it's double the requests?

No, for two reasons:

  1. monitor404 caches the result of a URL check for an hour, so heavily trafficked links don't get re-checked with every click.
  2. monitor404 only issues HEAD requests, which are significantly lighter than GET requests because they only return the HTTP headers and not the body content of the page.

Do all servers respond to HEAD requests with the same status code they respond to GET requests?

According to the W3 spec, they must. However, a few sites don't honor this and return 404 to all HEAD requests, or to all requests sent from an EC2 IP address (which monitor404 runs on). Consequently, monitor404 has had to ignore 404s from a few sites. An exhaustive list of ignored sites follows:

What happens if your server goes down?

monitor404 has graceful fallback measures in place to make sure all links work as normal if (heaven forbid) our server malfunctioned or went down.
